# FS — Functional Specification
## AI-планировщик отпусков производственного подразделения

> **Версия:** 1.0 (MVP)
> **Дата:** апрель 2026
> **Связанный документ:** URS v1.0

---

## 1. Назначение документа

Настоящий документ описывает **функциональные требования** к системе на уровне поведения — что именно делает система, как реагирует на действия пользователя, какие правила применяет. Каждое требование связано с пользовательским требованием из URS через поле `→ UR-XX`.

## 3. Модуль аутентификации и авторизации

### FR-AUTH-01 — Вход в систему `→ UR-01`
- Система предоставляет форму входа с полями «Логин» и «Пароль».
- При успешной аутентификации выдаётся JWT-токен.
- При неверных данных — сообщение «Неверный логин или пароль», без указания какое именно поле неверно.

### FR-AUTH-02 — Разграничение доступа (RBAC)
- Две роли: `EMPLOYEE`, `MANAGER`.
- Маршруты API защищены: эндпоинты доступны только носителям соответствующей роли.
- Сотрудник видит только свои данные; менеджер — данные всех цехов и все административные функции (структура, календарь, генерация, согласование).

## 4. Модуль структуры подразделения

### FR-STRUCT-01 — Иерархия подразделения `→ UR-12`
Иерархия для целей планирования и покрытия: **Цех → Смена**.

```
Цех (workshop)
  └── Смена (shift): Первая | Вторая | Третья | Круглосуточная
```

Линия / Участок хранится как **текстовое поле** у сотрудника (используется только для отображения в интерфейсе; в логику покрытия и ограничения AI не входит).

### FR-STRUCT-02 — Сотрудник `→ UR-12`
Каждый сотрудник имеет следующие атрибуты:

| Атрибут | Тип | Ограничения |
|---------|-----|-------------|
| ФИО | string | не пустое |
| Логин | string | уникальный |
| Пароль (хэш) | bcrypt | — |
| Роль в системе | enum | `EMPLOYEE / MANAGER` |
| Должность | string | свободный текст |
| Линия / Участок | string | свободный текст, необязательно (только для отображения) |
| Квалификация | enum | `KEY` (ключевой) / `STD` (взаимозаменяемый) |
| Смена | FK → Shift | обязательно; цех определяется через `shifts.workshop_id` |
| Норма дней отпуска | int | по умолчанию 28, редактируется менеджером |
| Дней использовано | int | вводится менеджером при создании (перенос остатка) |

### FR-STRUCT-03 — CRUD структуры
- Менеджер создаёт, редактирует и удаляет цеха, смены, сотрудников через интерфейс.
- Удаление возможно только при отсутствии зависимых данных (пожелания, отпускные блоки); при наличии зависимостей система выводит ошибку.

## 5. Модуль производственного календаря

### FR-CAL-01 — Типы периодов `→ UR-13`
Год разбивается на периоды. Каждый период имеет статус:

| Статус | Код | Поведение AI |
|--------|-----|--------------|
| Высокий сезон | `HIGH` | Отпуска запрещены ИЛИ ограничены квотой (настраивается) |
| Нейтральный | `NEUTRAL` | Без ограничений |
| Низкий сезон | `LOW` | Приоритетный период, AI старается размещать отпуска здесь |

### FR-CAL-02 — Интерфейс разметки `→ UR-13`
- Менеджер управляет периодами через простую форму-список: каждая строка содержит «Дата начала / Дата окончания / Статус / Квота (для `HIGH`)».
- Кнопка «Добавить период» добавляет новую строку; кнопка «Удалить» убирает выбранный период.
- Периоды не должны пересекаться; система проверяет это при сохранении.
- Разметка периодов отображается цветным фоном на Gantt-диаграмме.

### FR-CAL-03 — Минимальное покрытие `→ UR-14`
- Для каждого цеха менеджер задаёт **минимальное число присутствующих сотрудников** на каждый статус периода.
- Раздельно указывается минимум для квалификации `KEY` и суммарный минимум.
- Пример записи: `Цех №1 | HIGH | min_total=10 | min_key=2`.

### FR-CAL-04 — Государственные праздники
- Список нерабочих праздничных дней РФ на текущий год вшивается в систему как справочник.
- Праздники визуально выделяются на всех временных шкалах.
- При подсчёте календарных дней отпуска праздники **включаются** в счёт (по ТК РФ).

## 6. Модуль пожеланий сотрудников

### FR-WISH-01 — Форма пожеланий `→ UR-03`
- Сотрудник видит форму с тремя секциями (Вариант 1, Вариант 2, Вариант 3).
- Каждая секция содержит:
  - Дата начала (date picker)
  - Дата окончания (date picker)
  - Приоритет (выбирается автоматически по порядку: 1 = Вариант 1)
  - Комментарий (textarea, до 300 символов, необязательно)
- Варианты 2 и 3 необязательны.

### FR-WISH-02 — Баланс дней `→ UR-02`
- Над формой отображается статичная плашка:
  ```
  Норма: 28 дн.  |  Использовано: 0 дн.  |  Доступно: 28 дн.
  ```

### FR-WISH-03 — Валидация пожеланий `→ UR-04`
Пожелание — это предпочтение, а не финальный график. Соблюдение нормы ТК РФ (≥14 дней непрерывно) гарантируется AI на этапе генерации (HC-5) и не проверяется при подаче пожеланий.

Проверки выполняются при нажатии кнопки «Сохранить». Исключение: предупреждения о HIGH-сезоне показываются сразу после выбора дат.

| Проверка | Когда | Реакция |
|----------|-------|---------|
| Период пересекается с `HIGH` и квота = 0 | При выборе дат | Предупреждение под полем: «Высокий сезон — отпуска запрещены» |
| Период пересекается с `HIGH` и квота > 0 | При выборе дат | Информация: «Высокий сезон: одобрение ограничено» |
| Дата начала > дата окончания | При сохранении | Ошибка: «Некорректный период» |
| Запрошено дней > остаток | При сохранении | Ошибка: «Недостаточно дней отпуска» |

### FR-WISH-04 — Блокировка редактирования
- Пожелания можно редактировать до момента запуска AI-генерации.
- После запуска генерации форма переходит в режим «только чтение»; изменение возможно только через менеджера.

## 7. Модуль AI-генерации графика

### FR-AI-01 — Запуск генерации `→ UR-15`
- Менеджер нажимает кнопку «Сгенерировать график» — генерация запускается для всего подразделения на весь год.
- Генерация выполняется асинхронно; в интерфейсе отображается статус-бар с прогрессом.
- По завершении менеджер получает уведомление в интерфейсе (toast-уведомление + обновление страницы).

### FR-AI-02 — Входные данные оптимизатора
Оптимизатор (OR-Tools CP-SAT) получает следующие данные:

```
- Список сотрудников (квалификация, цех, смена, норма дней, остаток)
- Пожелания по каждому сотруднику (до 3 вариантов с приоритетами)
- Производственный календарь (периоды и их статусы)
- Минимальное покрытие по цехам (min_total, min_key на период)
- Список праздничных дней
- Правила ТК РФ (нормативный блок)
```

### FR-AI-03 — Жёсткие ограничения (нарушение = недопустимое решение)

| № | Ограничение |
|---|------------|
| HC-1 | Число присутствующих сотрудников цеха ≥ `min_total` в каждый день |
| HC-2 | Число присутствующих ключевых специалистов (`KEY`) ≥ `min_key` в каждый день |
| HC-3 | В период `HIGH` с квотой 0 — ни один сотрудник цеха не уходит в отпуск |
| HC-4 | В период `HIGH` с квотой N — не более N сотрудников цеха одновременно в отпуске |
| HC-5 | Каждому сотруднику назначается непрерывный отрезок отпуска ≥ 14 календарных дней (первая обязательная часть по ст. 125 ТК РФ) |
| HC-6 | Суммарная длительность отпуска сотрудника ≤ его норма дней (28 + доп.) |

### FR-AI-04 — Мягкие ограничения (нарушение = штраф в целевой функции)

| № | Условие | Штраф |
|---|---------|-------|
| SC-1 | Пожелание приоритета 1 не выполнено | 100 |
| SC-2 | Пожелание приоритета 2 не выполнено | 50 |
| SC-3 | Пожелание приоритета 3 не выполнено | 20 |
| SC-4 | Отпуск назначен в `HIGH`-период при наличии `LOW`-альтернативы | 80 |

**Целевая функция:** минимизация суммарного штрафа.

### FR-AI-05 — LLM-объяснения `→ UR-06, UR-11`
- После оптимизации для каждого сотрудника, чьё пожелание №1 не выполнено, LLM (OpenAI GPT-4o) генерирует объяснение.
- Промпт содержит: исходное пожелание, причину отказа (из данных оптимизатора), назначенный период.
- Объяснение — 2–4 предложения на русском языке, без технического жаргона.
- Пример вывода:
  > «Ваш приоритетный период (1–14 июля) не может быть одобрен: в это время в цехе №2 уже запланирован отпуск двух ключевых специалистов, что снизит покрытие ниже допустимого минимума. Вместо этого вам назначен период 5–18 сентября — он соответствует вашему второму варианту и приходится на рекомендованный для отпусков период.»

### FR-AI-06 — Альтернативные периоды `→ UR-11`
- При наличии конфликта LLM включает в текст объяснения **до 2 доступных альтернативных периодов** (первые подходящие слоты в `LOW` или `NEUTRAL`).
- Менеджер, при необходимости, вводит выбранный период вручную через «Изменить даты» (FR-APPR-02).
- Отдельный UI-компонент для выбора альтернатив не реализуется.

### FR-AI-07 — Результат генерации
- Результат сохраняется в БД как черновик (статус `DRAFT`).
- До утверждения руководителем график не виден сотрудникам.
- Можно запустить повторную генерацию — предыдущий черновик перезаписывается.

## 8. Модуль визуализации

### FR-VIS-01 — Gantt-диаграмма `→ UR-07`
- Ось X: временная шкала — весь год, фиксированный масштаб с горизонтальным скроллом.
- Ось Y: сотрудники, сгруппированные по `Цех → Смена`.
- Каждый отпускной блок окрашен по статусу:

| Статус блока | Цвет |
|---|---|
| `DRAFT` (черновик AI) | Серый |
| `PENDING` (на рассмотрении) | Жёлтый |
| `APPROVED` (утверждён) | Зелёный |
| `MODIFIED` (изменён менеджером) | Оранжевый |
| `CONFLICT` (критический конфликт) | Красный |

- На блоке отображается: ФИО (сокращённо) + число дней.
- Фон Gantt-диаграммы отражает производственный календарь: `HIGH` — светло-красный, `LOW` — светло-зелёный, `NEUTRAL` — белый.
- По клику на блок — всплывающая карточка с деталями и кнопками действий (для руководителя).

### FR-VIS-02 — Тепловая карта покрытия `→ UR-08`
- Отображается под Gantt-диаграммой или на отдельной вкладке.
- Строки — цеха; столбцы — недели года.
- Цвет ячейки по уровню покрытия:

| Уровень | Порог | Цвет |
|---------|-------|------|
| Норма | ≥ 100% от min_total | Зелёный (#22c55e) |
| На грани | 80–99% | Жёлтый (#eab308) |
| Дефицит | < 80% | Красный (#ef4444) |

- Цвет ячейки определяется по **худшему** из двух показателей: суммарного покрытия и покрытия по `KEY`.
- Наведение на ячейку — tooltip: «Неделя 23 | Цех №1 | Присутствует: 8/10 | KEY: 2/2».

### FR-VIS-03 — Личный кабинет сотрудника `→ UR-02, UR-05, UR-06`
- Баланс дней: плашка «Норма / Использовано / Запланировано / Остаток».
- Список поданных пожеланий с текущим статусом каждого (На рассмотрении / Утверждён / Изменён).
- Блок «Утверждённый отпуск»: даты текстом + объяснение AI (если пожелание было изменено).

## 9. Модуль конфликтов и предупреждений

### FR-CONF-01 — Типы конфликтов `→ UR-09`
Система автоматически выявляет и отображает:

| Код | Описание | Критичность |
|-----|----------|-------------|
| C-01 | Число присутствующих сотрудников цеха < `min_total` | Критический |
| C-02 | Число присутствующих `KEY` специалистов < `min_key` | Критический |
| C-03 | Часть отпуска сотрудника < 14 дней подряд | Критический |
| C-04 | Суммарные дни отпуска сотрудника > норма | Критический |
| C-05 | Отпуск в запрещённый `HIGH`-период | Критический |
| C-06 | Покрытие 80–99% от нормы | Предупреждение |

### FR-CONF-02 — Отображение конфликтов `→ UR-09, UR-16`
- На странице графика — панель «Конфликты» с количеством по каждому типу.
- Каждый конфликт раскрывается: затронутый сотрудник / цех / период + рекомендация AI.
- Критические конфликты (C-01 — C-05) блокируют кнопку «Утвердить» у конкретного блока.

## 10. Модуль согласования

### FR-APPR-01 — Жизненный цикл отпускного блока `→ UR-05, UR-10`

```
[AI_GENERATED] ──► [DRAFT] ──► [PENDING] ──► [APPROVED]
                                   │
                                   ├──► [MODIFIED]   (менеджер изменил даты)
                                   │
                                   └──► [CONFLICT]   (критический конфликт C-01—C-05)
```

> `DRAFT` — блок создан AI, но ещё не опубликован для согласования (до нажатия «Опубликовать»).
> `CONFLICT` — блок имеет активный критический конфликт; кнопка «Утвердить» заблокирована до его устранения.

### FR-APPR-02 — Действия менеджера `→ UR-10`
На странице согласования менеджер видит список блоков в статусе `PENDING`:

- **Утвердить** → статус меняется на `APPROVED`.
- **Изменить даты** → открывается форма с datepicker; при сохранении система повторно проверяет ТК и покрытие; статус → `MODIFIED`.

### FR-APPR-03 — Статус для сотрудника `→ UR-05`
- Сотрудник видит текущий статус в личном кабинете при загрузке страницы.
- При статусе `MODIFIED` — отображается объяснение от AI + комментарий менеджера (если введён).

## 11. Правила ТК РФ — нормативный блок

### FR-TK-01 — Обязательные проверки

| Норма | Статья ТК РФ | Реализация |
|-------|--------------|------------|
| Одна часть отпуска ≥ 14 кал. дней подряд | ст. 125 | Жёсткое ограничение HC-5 в оптимизаторе + валидация при ручной правке |
| Общий объём отпуска ≥ 28 кал. дней | ст. 115 | Норма хранится в профиле; AI не назначает меньше |
| Суммарный отпуск ≤ норма | — | Жёсткое ограничение HC-6 |
| Праздники входят в расчёт календарных дней | ст. 120 | Справочник праздников учитывается при подсчёте |

### FR-TK-02 — Блокировка утверждения
- Менеджер **не может** утвердить блок с активным критическим конфликтом C-03 или C-04.
- При попытке кнопка «Утвердить» неактивна; отображается: «Устраните нарушение ТК РФ перед утверждением».

## 12. Требования к данным и хранению

### FR-DATA-01 — Основные сущности БД (PostgreSQL)

```
users           — сотрудники (профиль, роль, квалификация, line_text, баланс дней)
workshops       — цеха
shifts          — смены
season_periods  — периоды производственного календаря
coverage_rules  — нормы покрытия по цеху/периоду
wish_requests   — пожелания сотрудников
vacation_blocks — сгенерированные/утверждённые отпускные блоки
generation_jobs  — статус асинхронной генерации (RUNNING | DONE | FAILED)
```

> Линия/Участок хранится как поле `line_text` в таблице `users`, отдельная таблица `lines` отсутствует.
> Конфликты вычисляются на лету из данных `vacation_blocks` и `coverage_rules` — отдельная таблица не требуется.
> Аудит-лог в MVP не реализуется.
