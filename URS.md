# URS — User Requirements Specification
## AI-планировщик отпусков производственного подразделения

> **Версия:** 1.0 (MVP)
> **Дата:** апрель 2026
> **Область применения:** Биотехнологическая / фармацевтическая компания, производственное подразделение с посменным графиком

---

## 1. Назначение документа

Настоящий документ описывает **пользовательские требования** к MVP веб-сервиса «AI-планировщик отпусков». Требования сформулированы с точки зрения конечных пользователей и отвечают на вопрос «что система должна позволять делать». Детальное поведение системы описано в FS (Functional Specification).

---

## 2. Роли пользователей

| Роль | Кто | Основная задача в системе |
|------|-----|--------------------------|
| **Сотрудник** | Рабочий/специалист цеха | Подать пожелания по отпуску, видеть результат |
| **Менеджер** | Начальник смены / цеха, HR | Настроить структуру и календарь, запустить AI, утвердить график, контролировать ТК |

> В MVP роли HR-менеджера и руководителя цеха объединены в одну роль **Менеджер**. Менеджер имеет доступ ко всем цехам.

---

## 3. Пользовательские требования

### 3.1 Сотрудник

| ID | Пользовательское требование |
|----|----------------------------|
| UR-01 | Сотрудник входит в систему по логину и паролю и видит только свои данные |
| UR-02 | Сотрудник видит текущий остаток дней отпуска (норма / использовано / доступно) |
| UR-03 | Сотрудник подаёт до 3 вариантов желаемого периода отпуска, указывая даты начала и окончания, приоритет (1–3) и необязательный комментарий |
| UR-04 | При подаче пожелания система немедленно предупреждает, если выбранный период попадает в запрещённый («высокий сезон») или превышает остаток дней |
| UR-05 | Сотрудник видит статус своего отпуска: **На рассмотрении / Утверждён / Изменён**; статус актуален на момент загрузки страницы |
| UR-06 | При изменении периода сотрудник получает понятное объяснение от AI — почему его пожелание было скорректировано и какой период предложен вместо |

---

### 3.2 Менеджер

| ID | Пользовательское требование |
|----|----------------------------|
| UR-07 | Менеджер видит Gantt-диаграмму отпусков по всем цехам на выбранный год — кто, когда и как долго в отпуске |
| UR-08 | Менеджер видит тепловую карту покрытия по неделям с цветовой индикацией: зелёный — норма, жёлтый — на грани, красный — дефицит |
| UR-09 | Система автоматически выделяет конфликты: недостаточное покрытие цеха, одновременный уход нескольких ключевых специалистов, нарушения ТК РФ |
| UR-10 | Менеджер утверждает отпускной период сотрудника или вручную изменяет даты; при изменении дат комментарий обязателен |
| UR-11 | Менеджер видит AI-объяснение по каждому конфликту и предлагаемое альтернативное решение |
| UR-12 | Менеджер добавляет, редактирует и удаляет цеха, смены, сотрудников с их ролью, квалификацией, привязкой к цеху/смене и необязательным указанием линии/участка |
| UR-13 | Менеджер размечает производственный календарь на год: отмечает «высокий сезон» (запрет/ограничение отпусков) и «низкий сезон» (рекомендуемый период) |
| UR-14 | Менеджер задаёт минимальное покрытие по каждому цеху и квалификации для каждого периода |
| UR-15 | Менеджер одним действием запускает AI-генерацию оптимального графика отпусков для всего подразделения на весь год |
| UR-16 | Менеджер видит сводный список конфликтов и нарушений ТК РФ по всем цехам до утверждения графика |
| UR-17 | Менеджер видит сводную таблицу: остатки дней по каждому сотруднику, статус согласования, наличие нарушений ТК |

## 4. Ключевые ограничения, принятые для MVP

1. **ТК РФ** — система обязана соблюдать: одна часть отпуска не менее 14 календарных дней подряд, общий объём не менее 28 дней в год.
2. **Производственное покрытие** — AI не вправе генерировать график, при котором число присутствующих сотрудников падает ниже заданного минимума.
3. **Сезонность** — в период «высокого сезона» отпуска не назначаются или ограничиваются установленной квотой.
4. **Квалификация** — не допускается одновременный уход всех ключевых специалистов (`KEY`) цеха ниже установленного минимума.
5. **Объяснимость** — каждое решение AI (перенос, ограничение) сопровождается объяснением на русском языке.

---

## 5. Вне области MVP

Следующие возможности **не входят** в MVP и реализуются в следующих версиях:

- Drag-and-drop на Gantt-диаграмме
- Email / push-уведомления и real-time обновление статусов
- Экспорт в PDF / XLSX / iCal
- Интеграция с 1С:ЗУП и корпоративным AD
- Telegram-бот
- AI-чат-ассистент
- Импорт сотрудников из CSV
- Отклонение (reject) отпускного блока с возвратом к пожеланиям
- Массовое утверждение (bulk approve)
- Фильтрация запуска генерации по цеху / кварталу
- Интерактивный timeline для разметки производственного календаря
- Аудит-лог (не хранится и не отображается)
- Переключение масштаба Gantt (год/квартал/месяц) — один фиксированный вид
- Динамический подсчёт дней в форме пожеланий
- Интерактивный UI выбора альтернативных периодов (альтернативы включены в текст AI-объяснения)
